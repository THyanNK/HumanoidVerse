# Humanoid 项目代码解释

本文档解释当前仓库的文件结构、主要文件职责，以及训练和评估时实际使用到的文件与代码流程。

## 1. 项目总体结构

```text
Humanoid/
├── eval.sh
├── run_humanoidverse_genesis_baseline.sh
├── run_humanoidverse_smoke.sh
├── run_humanoidverse_xvfb_video.sh
├── tools/
│   └── record_x11_display.py
├── HumanoidVerse/
│   ├── README.md
│   ├── pyproject.toml
│   ├── assets/
│   └── humanoidverse/
│       ├── train_agent.py
│       ├── eval_agent.py
│       ├── config/
│       ├── agents/
│       ├── envs/
│       ├── simulator/
│       ├── utils/
│       └── data/
├── btw/
├── HUMANOIDVERSE_REPRO.md
├── 实验.md
└── big_homework.png
```

这是一个基于 HumanoidVerse 的人形机器人强化学习项目。外层 `Humanoid/` 目录放实验脚本和说明文件，内层 `HumanoidVerse/` 是原始 Python 包主体。

## 2. 顶层文件说明

| 文件或目录 | 作用 |
|---|---|
| `eval.sh` | 当前推荐的评估入口。负责检查 checkpoint 和训练 `config.yaml`，启动 Xvfb，调用 `eval_agent.py`，同时用 `record_x11_display.py` 录屏。 |
| `run_humanoidverse_genesis_baseline.sh` | 完整 Genesis 训练入口。默认使用 4096 个环境、H1 10DoF、平地 locomotion 任务。 |
| `run_humanoidverse_smoke.sh` | 最小烟雾测试训练入口。默认 `num_envs=1`、训练 1 个 iteration，每个 iteration 2 步，用来快速确认环境能跑通。 |
| `run_humanoidverse_xvfb_video.sh` | 旧的 Xvfb 录屏训练入口。启动虚拟显示器，再运行 smoke 训练并录制 X11 根窗口。 |
| `tools/record_x11_display.py` | X11 录屏工具。按帧截取 X display root window，并用 `imageio` 写成 mp4。 |
| `HUMANOIDVERSE_REPRO.md` | 复现实验环境和命令说明。 |
| `实验.md` | 当前实验记录、报错日志和问题分析材料。 |
| `big_homework.png` | 课程作业相关图片资源。 |
| `btw/` | 额外参考 markdown 文档，不参与训练或评估主流程。 |
| `.gitignore` | Git 忽略规则。 |

## 3. HumanoidVerse 目录结构

### 3.1 包根目录

```text
HumanoidVerse/
├── README.md
├── LICENSE
├── pyproject.toml
├── assets/
└── humanoidverse/
```

| 文件或目录 | 作用 |
|---|---|
| `README.md` | 原项目说明，包含 IsaacGym、IsaacSim、Genesis 的环境安装和训练示例。 |
| `LICENSE` | 开源许可证。 |
| `pyproject.toml` | Python 包元数据和依赖列表，例如 `hydra-core`、`torch` 相关工具、`pynput`、`tensorboard`、`onnx` 等。 |
| `assets/` | README 展示用 GIF 和 logo，不参与训练逻辑。 |
| `humanoidverse/` | Python 包主体。训练、评估、环境、算法、仿真器、配置都在这里。 |

### 3.2 `humanoidverse/` 主体目录

```text
humanoidverse/
├── train_agent.py
├── eval_agent.py
├── config/
├── agents/
├── envs/
├── simulator/
├── utils/
└── data/
```

| 文件或目录 | 作用 |
|---|---|
| `train_agent.py` | 训练入口。读取 Hydra 配置，创建环境和 PPO 算法对象，保存训练配置，开始 `algo.learn()`。 |
| `eval_agent.py` | 评估入口。根据 checkpoint 找训练 `config.yaml`，创建环境和算法，加载模型，导出 ONNX，并进入无限评估循环。 |
| `config/` | Hydra YAML 配置。控制算法、环境、机器人、观测、奖励、地形、仿真器。 |
| `agents/` | 强化学习算法实现。当前主要是 PPO。 |
| `envs/` | 任务环境实现。当前使用 locomotion 任务，继承 legged base 环境。 |
| `simulator/` | 仿真后端适配层。包含 Genesis、IsaacGym、IsaacSim 三套接口。 |
| `utils/` | 配置预处理、数学、旋转、日志、观测解析、导出工具。 |
| `data/` | 机器人资产文件，包括 H1/G1 的 URDF、USD、XML 和 STL mesh。 |

## 4. 代码文件职责

### 4.1 训练和评估入口

| 文件 | 作用 |
|---|---|
| `humanoidverse/train_agent.py` | 训练主程序。Hydra 加载 `config/base.yaml`，根据命令行 override 组合算法、环境和仿真器配置。 |
| `humanoidverse/eval_agent.py` | 评估主程序。Hydra 加载 `config/base_eval.yaml`，再用 checkpoint 旁边的训练 `config.yaml` 补全训练配置。 |

### 4.2 算法模块

| 文件 | 作用 |
|---|---|
| `agents/base_algo/base_algo.py` | 算法基类，定义 `setup/learn/load/save` 等接口。 |
| `agents/ppo/ppo.py` | PPO 主实现。包含模型创建、rollout 收集、GAE return/advantage、PPO 更新、checkpoint 保存、评估循环。 |
| `agents/modules/ppo_modules.py` | Actor 和 Critic 网络包装。Actor 输出动作均值并维护高斯策略方差，Critic 输出 value。 |
| `agents/modules/modules.py` | 通用 MLP 构建器。根据 YAML 中的 `input_dim/output_dim/hidden_dims/activation` 创建网络。 |
| `agents/modules/data_utils.py` | `RolloutStorage`，保存每个 rollout step 的 observation、action、reward、done、value、advantage 等张量，并生成 mini-batch。 |
| `agents/callbacks/base_callback.py` | 评估 callback 基类。 |
| `agents/callbacks/analysis_plot_locomotion.py` | 评估/分析时的 locomotion 可视化或曲线服务。 |
| `agents/callbacks/analysis_plot_template.html` | 上述分析页面的 HTML 模板。 |

### 4.3 环境模块

| 文件 | 作用 |
|---|---|
| `envs/base_task/base_task.py` | 环境最底层基类。创建 simulator、地形、机器人资产、并提供 reset/render 等通用逻辑。 |
| `envs/legged_base_task/legged_robot_base.py` | 腿式机器人通用 RL 环境。负责 step 流程、PD 控制、奖励计算、终止条件、观测拼接、reset、domain randomization。 |
| `envs/locomotion/locomotion.py` | locomotion 任务特化。增加速度/朝向 command、tracking reward、脚部奖励、H1 locomotion 相关观测。 |
| `envs/env_utils/history_handler.py` | 保存历史观测，用于 history/short_history/long_history。 |
| `envs/env_utils/terrain.py` | 地形辅助逻辑。 |
| `envs/env_utils/command_generator.py` | 命令生成辅助模块。当前 H1 locomotion 主要在 `locomotion.py` 中直接采样 command。 |
| `envs/env_utils/general.py` | 通用对象转 dict 等小工具。 |
| `envs/env_utils/visualization.py` | 可视化点/线等辅助结构。 |

### 4.4 仿真器模块

| 文件 | 作用 |
|---|---|
| `simulator/base_simulator/base_simulator.py` | 仿真器抽象基类，规定 setup、terrain、load_assets、step、render 等接口。 |
| `simulator/genesis/genesis.py` | Genesis 后端适配。创建 `gs.Scene`，加载 URDF，读取/写入机器人状态，执行物理步进，刷新 viewer。当前训练主要使用此文件。 |
| `simulator/genesis/genesis_viewer.py` | Genesis viewer 控制占位类，当前多数方法为空。 |
| `simulator/genesis/tmp_gs_utils.py` | Genesis quaternion、坐标变换等临时工具函数。 |
| `simulator/isaacgym/isaacgym.py` | IsaacGym 后端适配。项目支持但当前实验命令不用。 |
| `simulator/isaacsim/isaacsim.py` | IsaacSim 后端适配。项目支持但当前实验命令不用。 |
| `simulator/isaacsim/*.py` | IsaacSim 的 articulation、event、viewpoint camera 等配置和工具。 |

## 6. 训练代码流程

### 6.1 `run_humanoidverse_genesis_baseline.sh`

这段脚本做四件事：

1. 定位项目根目录、HumanoidVerse 目录和 Python 环境。
2. 检查 `$ENV_DIR/bin/python` 是否存在。
3. 默认拒绝在 CPU 上跑 4096 环境的完整训练，避免误把服务器打满。
4. 进入 `HumanoidVerse/`，调用 `humanoidverse/train_agent.py`，并传入 Hydra override。

关键变量：

| 变量 | 默认值 | 作用 |
|---|---|---|
| `HUMANOIDVERSE_ENV` | `/inspire/.../.global_envs/humanoidverse` | 指定 conda/venv 环境路径。 |
| `PROJECT_NAME` | `HumanoidLocomotion` | 日志一级目录。 |
| `EXPERIMENT_NAME` | `H110dof_loco_Genesis` | run 名称的一部分。 |
| `NUM_ENVS` | `4096` | 并行仿真环境数。 |
| `HEADLESS` | `True` | 训练默认不打开 viewer。 |

### 6.2 `train_agent.py`

`train_agent.py` 的主要段落：

1. `@hydra.main(config_path="config", config_name="base")`
   - 以 `config/base.yaml` 为主配置。
   - 命令行 `+simulator=genesis` 等 override 会补齐 `base/structure.yaml` 中的必填项。

2. 判断 simulator 类型
   - 如果是 IsaacSim，启动 IsaacSim app。
   - 如果是 IsaacGym，先 import IsaacGym。
   - 当前命令是 Genesis，所以这两段基本跳过。

3. 初始化日志
   - Hydra 输出目录里写 `train.log`。
   - 控制台使用 `LOGURU_LEVEL`。

4. `pre_process_config(config)`
   - 来自 `utils/helpers.py`。
   - 根据 `obs` 配置计算 `actor_obs` 和 `critic_obs` 维度。
   - 当前 H1 10DoF 单步观测维度为：
     - base_lin_vel: 3
     - base_ang_vel: 3
     - projected_gravity: 3
     - command_lin_vel: 2
     - command_ang_vel: 1
     - dof_pos: 10
     - dof_vel: 10
     - actions: 10
     - 总计 42

5. 创建环境
   - `env = instantiate(config=config.env, device=device)`
   - 当前对应 `LeggedRobotLocomotion`。

6. 保存训练配置
   - 保存到 `logs/HumanoidLocomotion/<timestamp>-H110dof_loco_Genesis-locomotion-h1_10dof/config.yaml`。
   - eval 必须依赖这个文件恢复训练配置。

7. 创建 PPO
   - `algo = instantiate(config.algo, env=env, device=device, log_dir=experiment_save_dir)`
   - 对应 `agents/ppo/ppo.py`。

8. 加载 checkpoint 并训练
   - `algo.load(config.checkpoint)`：如果 checkpoint 为 null 就从头训练。
   - `algo.learn()`：进入 PPO 训练循环。

### 6.3 环境创建流程

创建 `LeggedRobotLocomotion` 时的继承链：

```text
LeggedRobotLocomotion
└── LeggedRobotBase
    └── BaseTask
```

`BaseTask.__init__` 做基础搭建：

1. 根据 `config.simulator._target_` 创建仿真器。
2. 当前是 `humanoidverse.simulator.genesis.genesis.Genesis`。
3. 设置 headless。
4. 调用 `simulator.setup()` 创建 Genesis scene。
5. 调用 `simulator.setup_terrain()` 添加平地。
6. 调用 `simulator.load_assets()` 加载 H1 URDF。
7. 创建 env origins。
8. `simulator.create_envs()` 构建并行环境。
9. 读取 DOF 限制、刚体索引、初始状态。
10. 非 headless 时初始化 viewer。

`LeggedRobotBase.__init__` 补充腿式机器人 RL 逻辑：

1. 初始化动作、力矩、PD gain、历史观测、contact buffer。
2. 根据 reward 配置准备 reward 函数列表。
3. 初始化 domain randomization buffer。

`LeggedRobotLocomotion.__init__` 补充 locomotion 任务：

1. 创建 command buffer：`commands[num_envs, 4]`。
2. 设置线速度、横向速度、yaw 角速度、heading 的采样范围。
3. 提供 locomotion 专属 reward 和 command 观测。

### 6.4 Genesis 仿真器流程

`simulator/genesis/genesis.py` 是当前训练的仿真后端。

主要代码段：

1. `__init__`
   - 保存配置和设备。
   - 调用 `gs.init`，有 CUDA 时使用 Genesis GPU backend。

2. `setup`
   - 从 `config/simulator/genesis.yaml` 读取：
     - `fps=200`
     - `control_decimation=4`
     - `substeps=1`
   - 创建 `gs.Scene`。
   - 非 headless 时创建 viewer。
   - headless 时关闭 visualizer 的 rasterizer/renderer，避免服务器无 EGL 时出问题。

3. `setup_terrain`
   - 当前 `mesh_type=plane`，添加 `gs.morphs.Plane()`。

4. `load_assets`
   - 根据 `robot.asset.asset_root` 和 `robot.asset.urdf_file` 找到 `h1/h1_10dof.urdf`。
   - 用 `gs.morphs.URDF` 加载机器人。
   - 记录 DOF 名称、body 名称和 joint index。

5. `create_envs`
   - 调用 `self.scene.build(n_envs=num_envs)` 构建并行环境。

6. `prepare_sim` / `refresh_sim_tensors`
   - 从 Genesis 读取 base pose、base velocity、joint position、joint velocity、contact force、刚体位置姿态速度。

7. `apply_torques_at_dof`
   - 把 PPO 动作经过 PD 控制后得到的 torque 发送给 Genesis。

8. `simulate_at_each_physics_step`
   - 每个物理子步调用 `self.scene.step()`。

9. `render`
   - 非 headless 时刷新 Genesis visualizer/viewer。
   - 这是为 Xvfb 录屏补的关键逻辑，否则 viewer 可能创建了但画面不刷新。

### 6.5 每一步环境交互

PPO 调用：

```python
obs_dict, rewards, dones, infos = self.env.step(actor_state)
```

实际进入 `LeggedRobotBase.step`：

1. `_pre_physics_step(actions)`
   - 裁剪 action。
   - 如果启用控制延迟，则写入 action queue。

2. `_physics_step()`
   - 非 headless 时先 render。
   - 重复 `control_decimation` 次：
     - `_compute_torques(actions_after_delay)`
     - `simulator.apply_torques_at_dof(torques)`
     - `simulator.simulate_at_each_physics_step()`

3. `_post_physics_step()`
   - 刷新仿真张量。
   - 更新 episode length 和计数器。
   - locomotion 中更新 command。
   - 检查终止条件。
   - 计算 reward。
   - reset 需要 reset 的环境。
   - 计算 observation。
   - 写入 history。
   - 返回 obs/reward/done/info。

### 6.6 PPO 训练流程

`PPO.learn()`：

1. `env.reset_all()` 获取初始 obs。
2. 每个 iteration 执行：
   - `_rollout_step(obs_dict)`
   - `_training_step()`
   - `_post_epoch_logging()`
   - 按 `save_interval` 保存 `model_<it>.pt`
3. 训练结束再保存最终模型。

`_rollout_step`：

1. 循环 `num_steps_per_env` 次。
2. actor 根据 `actor_obs` 采样 action。
3. critic 根据 `critic_obs` 估计 value。
4. 将 obs/action/value/log_prob 等写入 `RolloutStorage`。
5. 环境 step，得到下一个 obs、reward、done。
6. 计算 GAE returns 和 advantages。

`_training_step`：

1. `RolloutStorage.mini_batch_generator` 生成 mini-batch。
2. `_update_ppo` 计算 PPO loss：
   - policy surrogate loss
   - clipped value loss
   - entropy bonus
   - adaptive KL learning rate
3. actor/critic 分别反向传播和优化。

### 6.7 Actor/Critic 网络

当前 PPO 配置来自 `config/algo/ppo.yaml`：

```yaml
actor hidden_dims: [512, 256, 128]
critic hidden_dims: [512, 256, 128]
activation: ELU
init_noise_std: 0.8
```

`PPOActor`：

1. 用 `BaseModule` 创建 MLP。
2. 输出 action mean。
3. 用可学习参数 `std` 构造 Normal 分布。
4. 训练时 `act` 采样动作。
5. eval 时 `act_inference` 直接输出均值动作。

`PPOCritic`：

1. 用 `BaseModule` 创建 MLP。
2. 输入 `critic_obs`。
3. 输出 state value。

### 7.1 `eval.sh`

`eval.sh` 做这些事情：

1. 固定使用环境：

```bash
ENV_DIR="${HUMANOIDVERSE_ENV:-/inspire/qb-ilm/project/robot-reasoning/public/cyh/.global_envs/humanoidverse}"
PYTHON_BIN="${PYTHON_BIN:-$ENV_DIR/bin/python}"
```

1. 解析 checkpoint：
   - 支持 `bash eval.sh path/to/model.pt`
   - 支持 `CHECKPOINT=path/to/model.pt bash eval.sh`
   - 支持 `+checkpoint=path/to/model.pt`

2. 在启动 Python 前检查：
   - checkpoint 文件存在。
   - checkpoint 同目录或上一级目录有 `config.yaml`。

3. 启动 Xvfb：

```bash
Xvfb ":$DISPLAY_NUM" -screen 0 "$XVFB_SCREEN" -nolisten tcp
export DISPLAY=":$DISPLAY_NUM"
```

1. 启动 eval：

```bash
$PYTHON_BIN humanoidverse/eval_agent.py +checkpoint=... +headless=False
```

1. 启动录屏：

```bash
$PYTHON_BIN tools/record_x11_display.py --pid "$EVAL_PID" --output "$VIDEO_OUTPUT" --fps "$VIDEO_FPS" --delay "$VIDEO_DELAY" --max-seconds "$VIDEO_MAX_SECONDS"
```

1. 录制结束后杀掉 eval 进程。

### 7.2 `record_x11_display.py`

主要段落：

1. 参数解析：
   - `--output` 输出 mp4。
   - `--pid` 录到某个进程结束为止。
   - `--fps` 帧率。
   - `--delay` 录制前等待。
   - `--max-seconds` 最大录制时长。
   - `--crop` 可选裁剪区域。

2. 连接 X11：

```python
xdisplay = display.Display()
root = xdisplay.screen().root
```

1. 抓帧：

```python
root.get_image(...)
Image.frombytes(...)
writer.append_data(frame)
```

1. 写 mp4：

```python
imageio.get_writer(str(output), fps=args.fps)
```

### 7.3 `eval_agent.py`

主要段落：

1. 键盘监听相关：
   - `on_press` 支持按键切任务或施加手部 force。
   - `listen_for_keypress` 延迟 import `pynput`。
   - `should_enable_keyboard_listener` 默认关闭键盘监听，避免服务器没有 DISPLAY 时崩溃。

2. `@hydra.main(config_path="config", config_name="base_eval")`
   - eval 本身先加载 `base_eval.yaml`。
   - `base_eval.yaml` 不包含 simulator/env/robot/algo，所以必须从 checkpoint 找训练 `config.yaml`。

3. checkpoint 配置恢复：

```python
config_path = checkpoint.parent / "config.yaml"
if not exists:
    config_path = checkpoint.parent.parent / "config.yaml"
```

如果找不到，后面会缺少 `config.simulator`，这就是之前 `Key 'simulator' is not in struct` 的根因。

1. 合并 eval overrides：
   - 训练配置里的 `eval_overrides` 会把 `headless=False`、`num_envs=1` 等评估参数合并进去。
   - 命令行参数优先级最高。

2. 创建环境和算法：
   - `pre_process_config(config)`
   - `env = instantiate(config.env, device=device)`
   - `algo = instantiate(config.algo, env=env, device=device, log_dir=None)`
   - `algo.setup()`
   - `algo.load(config.checkpoint)`

3. 导出 ONNX：
   - 当前 `EXPORT_ONNX=True`。
   - 先用 `algo.get_example_obs()` 获取样例 obs。
   - 再导出 actor/critic。

4. 无限评估循环：

```python
algo.evaluate_policy()
```

`PPO.evaluate_policy()` 内部是 `while True`，所以 `eval.sh` 通过录制固定时长后杀进程来结束。

## 10. 常见问题和对应文件

| 问题 | 相关文件 | 原因/处理 |
|---|---|---|
| `pynput` 报 DISPLAY 错误 | `eval_agent.py` | 服务器没有 X11。当前已改成延迟导入，并默认关闭键盘监听。 |
| eval 报 `Key 'simulator' is not in struct` | `eval_agent.py`、checkpoint 目录 | checkpoint 旁边没有训练 `config.yaml`。 |
| 录出来是黑屏 | `genesis.py`、`record_x11_display.py`、`eval.sh` | Xvfb 只是虚拟显示器，必须保证 Genesis viewer 刷帧；当前已在 Genesis `render()` 中补刷新兜底。 |
| eval 不会自动结束 | `agents/ppo/ppo.py` | `evaluate_policy()` 是 `while True`，所以由 `eval.sh` 控制录制时长后杀进程。 |
| 相对 checkpoint 找不到 | `eval.sh` | 相对路径按 `HumanoidVerse/` 目录解析。运行时看脚本打印的 `Checkpoint path`。 |
