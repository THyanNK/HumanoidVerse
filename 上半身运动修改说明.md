# 上半身运动修改说明

## 结论

当前只能让下半身动，不是因为策略偶然没有学会上半身动作，而是因为当前实验使用的是 `h1_10dof` 机器人配置和 `h1_10dof.urdf`。这条配置链路从物理模型、动作维度、观测维度、可控关节列表到 PPO 网络输出都只暴露了 10 个下半身关节；上半身关节在 URDF 中还被定义成 `fixed`，物理上不能转动。

要让上半身也动起来，推荐不要在 `h1_10dof.yaml` 上硬改 `actions_dim`，而是切换到项目已有的完整 H1 配置 `h1.yaml` 和 `h1.urdf`，重新训练一个 19DoF 策略，再配合上半身自然摆臂/稳定躯干的奖励设计。

## 限制上半身运动的位置

### 1. 训练脚本选择了 10DoF 机器人

当前训练入口里显式使用了 10DoF 配置：

- `run_humanoidverse_genesis_baseline.sh:63`
- `run_humanoidverse_smoke.sh:53`

关键参数是：

```bash
+robot=h1/h1_10dof
```

这会加载 `HumanoidVerse/humanoidverse/config/robot/h1/h1_10dof.yaml`，后续动作空间、观测空间、URDF 路径都会跟着这个配置走。

### 2. `h1_10dof.yaml` 只定义 10 个动作和 10 个可观测 DOF

文件：`HumanoidVerse/humanoidverse/config/robot/h1/h1_10dof.yaml`

关键限制：

```yaml
num_bodies: 11
dof_obs_size: 10
actions_dim: 10
has_upper_body_dof: False
```

`dof_names` 只包含 10 个下半身关节：

- `left_hip_yaw_joint`
- `left_hip_roll_joint`
- `left_hip_pitch_joint`
- `left_knee_joint`
- `left_ankle_joint`
- `right_hip_yaw_joint`
- `right_hip_roll_joint`
- `right_hip_pitch_joint`
- `right_knee_joint`
- `right_ankle_joint`

`body_names` 也只保留 pelvis 和左右腿 link，没有把 torso、shoulder、elbow 等上半身 body 放进当前可用 body 列表。

这意味着：策略输出只有 10 维动作，环境也只会把这 10 维动作映射到下半身关节。

### 3. `h1_10dof.urdf` 把上半身关节定义成固定关节

文件：`HumanoidVerse/humanoidverse/data/robots/h1/h1_10dof.urdf`

这些上半身关节都是 `type="fixed"`：

- `torso_joint`
- `left_shoulder_pitch_joint`
- `left_shoulder_roll_joint`
- `left_shoulder_yaw_joint`
- `left_elbow_joint`
- `right_shoulder_pitch_joint`
- `right_shoulder_roll_joint`
- `right_shoulder_yaw_joint`
- `right_elbow_joint`

所以即使后面强行把动作维度改大，只要 URDF 还是 `h1_10dof.urdf`，这些关节在物理仿真里仍然不能转动。

对比项目已有的完整模型 `HumanoidVerse/humanoidverse/data/robots/h1/h1.urdf`，同名上半身关节是 `type="revolute"`，也就是可转动关节。

### 4. Genesis 加载器只控制配置中列出的 DOF

文件：`HumanoidVerse/humanoidverse/simulator/genesis/genesis.py`

`load_assets()` 里按配置加载 URDF：

```python
asset_file = self.robot_cfg.asset.urdf_file
asset_path = os.path.join(asset_root, asset_file)
```

然后只保留配置里的 body：

```python
links_to_keep=self.robot_cfg.body_names
```

最后只把配置里的 `dof_names` 转成可控 joint id：

```python
dof_names_list = copy.deepcopy(self.robot_cfg.dof_names)
self.dof_ids = [
    self.robot.get_joint(name).dof_idx_local
    for name in dof_names_list
]
self.num_dof = len(dof_names_list)
```

因此，真正进入仿真的可控 DOF 完全由 robot yaml 的 `dof_names` 决定。当前 `h1_10dof.yaml` 没有列出上半身关节，Genesis 就不会给策略控制这些关节。

### 5. 环境要求 DOF 数和动作维度一致

文件：`HumanoidVerse/humanoidverse/envs/base_task/base_task.py`

环境初始化时会检查：

```python
assert self.num_dof == self.dim_actions, "Number of DOFs must be equal to number of actions"
```

所以不能只改一个地方。若要改成全身控制，必须同时保证：

- `dof_names` 长度是 19
- `actions_dim` 是 19
- `dof_obs_size` 是 19
- `default_joint_angles` 有 19 个关节
- 关节上下限、速度限制、力矩限制都是 19 维
- PD 参数能匹配 torso、shoulder、elbow 等关节

### 6. PPO 网络输出维度来自 `robot.actions_dim`

文件：`HumanoidVerse/humanoidverse/agents/ppo/ppo.py`

PPO 初始化时使用：

```python
self.num_act = self.env.config.robot.actions_dim
```

然后创建 actor：

```python
PPOActor(..., num_actions=self.num_act, ...)
```

当前 10DoF checkpoint 的 actor 输出层是 10 维。切换到完整 H1 后 actor 输出会变成 19 维，因此旧的 `h1_10dof` checkpoint 不能直接作为完整上半身控制策略使用。通常需要重新训练，或者做谨慎的部分参数迁移。

### 7. eval 会读取 checkpoint 旁边保存的训练配置

文件：`HumanoidVerse/humanoidverse/eval_agent.py`

评估时如果传入 checkpoint，会先找：

```python
config_path = checkpoint.parent / "config.yaml"
```

找不到再找：

```python
config_path = checkpoint.parent.parent / "config.yaml"
```

然后加载训练时保存的配置：

```python
train_config = OmegaConf.load(file)
```

所以如果 checkpoint 来自 `h1_10dof` 训练日志，即使你在命令行临时改了机器人参数，eval 仍然会优先使用旧日志里的 `config.yaml`。要评估上半身可动的策略，必须使用用完整 H1 配置训练出来的新 checkpoint。

## 项目里已有的完整 H1 配置

文件：`HumanoidVerse/humanoidverse/config/robot/h1/h1.yaml`

这个文件已经基本具备全身控制需要的配置：

```yaml
num_bodies: 20
dof_obs_size: 19
actions_dim: 19
has_upper_body_dof: True
```

它的 `dof_names` 包含 19 个关节：

- 10 个下半身关节
- `torso_joint`
- 左臂 4 个关节：`left_shoulder_pitch_joint`、`left_shoulder_roll_joint`、`left_shoulder_yaw_joint`、`left_elbow_joint`
- 右臂 4 个关节：`right_shoulder_pitch_joint`、`right_shoulder_roll_joint`、`right_shoulder_yaw_joint`、`right_elbow_joint`

它还定义了：

- `upper_dof_names`
- `lower_dof_names`
- 20 个 `body_names`
- 完整 H1 资产路径，`robot_type: h1`
- `urdf_file: h1/${robot.asset.robot_type}.urdf`，实际会指向 `h1/h1.urdf`
- torso、shoulder、elbow 的 PD stiffness/damping
- 19 个关节的默认角度、位置限制、速度限制、力矩限制

因此后续修改的首选方向是直接用 `+robot=h1/h1`，而不是手工把 `h1_10dof` 扩成 19DoF。

## 推荐修改路线

### 第一步：先切换到完整 H1 配置做最小验证

后续训练命令里的机器人参数应从：

```bash
+robot=h1/h1_10dof
```

改为：

```bash
+robot=h1/h1
```

这会让系统加载：

- `HumanoidVerse/humanoidverse/config/robot/h1/h1.yaml`
- `HumanoidVerse/humanoidverse/data/robots/h1/h1.urdf`

预期变化：

- 动作维度从 10 变成 19
- `dof_pos`、`dof_vel`、`actions` 相关观测从 10 维变成 19 维
- actor 观测维度会随 `${robot.dof_obs_size}` 自动变大
- PPO actor 输出层从 10 维变成 19 维

建议先用很小的 `num_envs`、短时间 smoke run 验证完整 H1 能否在 Genesis 里正常加载、reset 和 step，不要一开始就跑完整训练。

### 第二步：不要复用旧 10DoF checkpoint 直接评估

旧日志路径中带有 `h1_10dof` 的 checkpoint 对应的是 10 维动作网络，并且日志里的 `config.yaml` 也会记录 `+robot=h1/h1_10dof`。

因此：

- 旧 checkpoint 可以作为下半身 baseline 结果展示
- 不能指望旧 checkpoint 让上半身动
- 训练完整 H1 后，要用新日志目录里的 `model_xxx.pt` 做 eval

如果一定想利用旧策略，只能做部分迁移：复用前面的特征层或下半身相关参数，新建 19 维输出层。但这需要专门写 load 逻辑，不能直接 `algo.load(old_10dof_checkpoint)`。

### 第三步：调整奖励，避免上半身乱动或僵硬

当前 H1 locomotion 奖励文件是：

`HumanoidVerse/humanoidverse/config/rewards/loco/reward_h1_locomotion.yaml`

它主要包含速度跟踪、姿态、力矩、动作变化率、脚部高度等 locomotion 奖励。代码里还有一个上半身冻结奖励函数：

文件：`HumanoidVerse/humanoidverse/envs/locomotion/locomotion.py`

```python
def _reward_upperbody_joint_angle_freeze(self):
    assert self.config.robot.has_upper_body_dof
```

这个奖励的作用是惩罚上半身偏离默认关节角。如果把它加到 `reward_scales` 并给较大的负权重，上半身会倾向于保持不动。要让上半身自然摆动，不应该强力启用这个 freeze 奖励。

更合理的奖励方向是：

- 保留速度跟踪：`tracking_lin_vel`、`tracking_ang_vel`
- 保留稳定性：`penalty_orientation`、`penalty_ang_vel_xy`、`base_height`
- 保留能耗和平滑性：`penalty_torques`、`penalty_dof_acc`、`penalty_action_rate`
- 对上半身增加轻量规则，而不是完全冻结

建议新增或改造的上半身奖励：

1. 上半身动作平滑：惩罚 upper DOF 的 action rate 和 acceleration，避免手臂抖动。
2. 肩肘关节幅度限制：允许摆臂，但限制 shoulder/elbow 进入奇怪姿态。
3. 左右臂对称性：左右肩 pitch 可以反相摆动，roll/yaw 不要过大。
4. 手臂和腿的相位关系：左腿向前时右臂向前，右腿向前时左臂向前。
5. 躯干稳定：torso 可以小幅参与平衡，但不要大幅前后左右摆。
6. 如果有 motion capture/reference motion，优先加入上半身 imitation reward，这比手写相位规则更容易得到自然行走。

### 第四步：分阶段训练

建议训练流程：

1. 加载 `h1.yaml` 和 `h1.urdf`，先验证仿真不报错。
2. 第一阶段让完整 H1 学会稳定站立和低速行走，上半身动作惩罚稍强，避免训练初期摔倒。
3. 第二阶段减弱上半身冻结或姿态保持惩罚，让 shoulder/elbow 可以随步态摆动。
4. 第三阶段加入摆臂自然性奖励或参考动作 imitation，让上半身动作和腿部步态同步。
5. 最后再打开更强的 domain randomization，提高泛化和鲁棒性。

这种方式比一开始直接放开 19 个 DOF 更稳，因为上半身多出来的 9 个自由度会显著扩大动作空间，训练初期更容易出现乱摆、摔倒、能耗变高。

## 不推荐的改法

### 不推荐只改 `actions_dim: 19`

如果只把 `h1_10dof.yaml` 的 `actions_dim` 改成 19，会导致环境断言失败或动作和 DOF 对不上。因为 `dof_names` 仍然只有 10 个。

### 不推荐只往 `dof_names` 加上半身关节

如果 URDF 还是 `h1_10dof.urdf`，上半身关节是 `fixed`。固定关节没有可控 DOF，Genesis 里 `get_joint(name).dof_idx_local` 可能不存在或无法作为正常可控关节使用。

### 不推荐直接改 `h1_10dof.urdf` 的 joint type

项目已经有完整的 `h1.urdf` 和 `h1.yaml`。直接修改 10DoF URDF 容易漏掉 joint limit、effort、velocity、body list、motion retarget 配置等配套内容。除非后面有特殊需求，否则优先使用已有完整 H1 配置。

## 后续真正要改的文件清单

如果开始动代码，建议按这个顺序改：

1. 训练脚本  
   把 `run_humanoidverse_genesis_baseline.sh` 和需要使用的 smoke/eval 脚本中的 `+robot=h1/h1_10dof` 改为 `+robot=h1/h1`。

2. 奖励配置  
   在 `HumanoidVerse/humanoidverse/config/rewards/loco/reward_h1_locomotion.yaml` 中增加上半身自然摆动相关 reward scale，或者先确保没有强力启用 `upperbody_joint_angle_freeze`。

3. 奖励函数  
   在 `HumanoidVerse/humanoidverse/envs/locomotion/locomotion.py` 中新增 upper-body smoothness、arm symmetry、arm-leg phase 等奖励函数。

4. 训练参数  
   完整 H1 动作空间更大，建议先减小训练规模或缩短 smoke 验证时间；正式训练时可能需要更多迭代。

5. eval  
   使用完整 H1 新训练出来的 checkpoint。不要用旧 `h1_10dof` 日志里的 `model_400.pt` 来判断上半身是否能动。

## 判断修改是否成功

成功切到完整 H1 后，日志里应该能看到：

- URDF 路径从 `h1_10dof.urdf` 变成 `h1.urdf`
- robot config 里 `actions_dim` 是 19
- robot config 里 `dof_obs_size` 是 19
- `has_upper_body_dof` 是 `True`
- checkpoint 日志目录名不再是 `h1_10dof`
- eval 读取的新 `config.yaml` 中 robot 是 `h1/h1`

如果录屏里上半身仍然不动，需要继续检查：

1. eval 是否仍然加载旧 checkpoint 的旧 `config.yaml`。
2. 训练脚本是否还有某处覆盖了 `+robot=h1/h1_10dof`。
3. reward 是否启用了强力的 `upperbody_joint_angle_freeze`。
4. PPO 是否实际加载了 19 维动作模型，而不是 10 维旧模型。
5. Genesis 日志中的 URDF 是否确实是 `h1.urdf`。
